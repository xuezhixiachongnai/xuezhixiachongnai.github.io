这里总结一下最近在实习中用到的 Git 命令。

## git add 

如果你添加了一个文件，或者是对原来的文件进行修改。要想将这些文件继续由 `git` 管理的话就需要执行这个命令。

我最常使用的是 `git add .` 一把梭哈。将当前所有发生变化的文件全部添加到暂存区。

> 这里涉及到了一个概念 **暂存区**。当我们的代码没有被版本仓库管理时，代码其实是在 **工作区** 的。工作区就是我们可以随意修改代码的地方。当完成修改执行 `git add .` 后。我们的代码就提交到暂存区了。暂存区是用来存放没有被提交到分支的地方。

当然，如果你想要添加某一个文件，或者某一个目录的文件，可以执行 `git add file-path`。这里 `file-path` 就是文件路径。上面的 `.` 其实就是代表当前目录，所以本质上没有什么变化。

我们修改文件后，可以在 `git add .` 前后执行 `git status` 试试，`git status` 是用来查看当前仓库状况的。

## git commit

代码肯定不能一直在暂存区待着，肯定是要提交到自己的分支的。只有把代码提交到自己的分支上，之后切换分支，或者合并远程代码是才舒坦，要不然无法执行切换分支等的一些命令。在学习 `git commit` 之前，先简单补一下其他操作。

### git init

我们要想通过 Git 管理我们的代码，首先要执行 `git init` 指令。这时，当前目录就是整个版本仓库，Git 只能管理当前目录，这时进行的任何文件操作都会被 Git 追踪。

### git branch

执行完 `git init` 之后，我们可以执行一下 `git branch --show-curent`，该命令会显示当前所处分支。刚刚初始化的仓库，`HEAD` 会指定一个默认名，因此执行完命令后很可能会显示 `master` 或者 `main`。

上面是显示一个分支，如果我们想要列出仓库的所有分支，可以执行 `git branch`。

> 需要注意，`git branch` 只会列出已经提交过内容的分支。当我们刚 `git init` 一个仓库时，没有任何提交，因此执行 `git branch` 时什么都不会显示。

### git switch

如果我们想要在其他分支开发，这时我们要如何创建分支呢？可以执行 `git switch -c dev-han`。

执行上述命令后，你就会有一个 `dev-han` 分支了！并且会自动切换到刚创建的分支。当然，如果你不喜欢 `dev-han` 这个名字，你可以自由命名。

如果你不想切换分支，只是想创建，那么可以执行 `git branch dev-han`；如果只想切换到别的分支，可以执行 `git switch master`。

> 如果我们的仓库没有任何的提交，执行创建和切换命令时大概率会报错。。。
>
> 下面给出 ChatGPT 选手的回答：
>
> #### P：为什么没有有提交记录的分支，不能执行上述操作
>
> 因为在 Git 里，分支本质不是一个独立的东西，它就是一个**指向某个提交（commit）的指针/引用**。
>
> 分支到底是什么？
>
> - 分支名（比如 `master`、`dev-han`）对应的是一个引用：`refs/heads/dev-han`
> - 这个引用里面存的是一个 **commit 的哈希**（指向哪个提交）
>
> 有提交时是这样的：
>
> ```markdown
> dev-han ->  a1b2c3d (commit)
> master  ->  9e8f7g6 (commit)
> HEAD    ->  dev-han
> ```
>
> 如果没有任何提交，比如刚初始化仓库时：
>
> - **仓库里一个 commit 都没有**
> - 那就意味着：根本没有 commit 对象可以让分支去指
>
> 所以此时你看到的 `master` / `dev-han`，很多情况下只是：
>
> - `HEAD` 里记着一个我当前分支叫这个名字的 **符号引用**
> - 但 `refs/heads/master`、`refs/heads/dev-han` 这些真正的分支指针可能还不存在，或存在但不指向任何 commit（unborn branch）
>
> ##### 1）`git switch master` 为什么失败？
>
> `git switch` 需要切到一个 **可解析的引用**（能找到目标 commit）。但 `master` 在零提交时通常并不存在为一个可用引用（或无法解析到 commit），所以报：`invalid reference: master`
>
> ##### 2）`git branch fix-han` 为什么失败？
>
> `git branch fix-han` 的默认语义是：从当前 `HEAD` 指向的提交点，分出一个新分支
>
> 可你现在的 `HEAD` 并没有指向任何 commit（因为没有 commit），所以 Git 没有起点，就会报：
> `not a valid object name: 'dev-han'`
>
> **没有提交 = 没有 commit 对象 = 分支没有目标可指**。于是很多操作（切换到某分支、从当前分支再创建新分支、比较分支差异等）都无法成立。
>
> #### 为什么 git switch -c 就可以
>
> 因为 `git switch -c dev-han` 在零提交仓库里做的是一件 **更轻** 的事：它只需要把 **HEAD 的当前分支名改掉**，并不一定要马上切到一个真实存在的分支引用/提交。
>
> ##### 1）`git switch -c dev-han` 在无提交时能做什么
>
> 在没有任何 commit 的仓库里，Git 允许你把 `HEAD` 从
>
> - 我叫 master（符号引用）
>   改成
> - 我叫 dev-han（符号引用）
>
> 这一步 **不需要 commit**，所以能成功。
>
> ##### 2）那为什么 `git switch master` 会失败？
>
> `git switch master` 的语义是：**切换到一个已经存在的分支/引用**。
>
> 但你现在这个仓库里：
>
> - `master` 这个 `refs/heads/master` 很可能 **根本没创建**
> - 或者即便有名字，也无法解析到任何 commit（unborn）
>
> 所以 Git 说：我找不到一个有效的 `master` 可以切过去 → `invalid reference: master`
>
> ##### 3）为什么 `git branch fix-han` 也会失败？
>
> `git branch fix-han` 默认等价于：从当前 `HEAD` 指向的提交分叉出一个新分支
>
> 可现在 `dev-han` 并没有指向任何 commit（仓库 0 提交），Git 没有起点 commit，所以报： `not a valid object name: 'dev-han'`

看了上面那么多内容，是不是发现没有说 `HEAD` 是什么？

我们其实可以把 `HEAD` 理解为是当前所在分支的一个 commit 上的指针。

恶补完上面的知识后，现在来看 `git commit`。

如果暂存区有内容，我们执行 `git commit -m "..."`。该命令会将当前暂存区的代码提交。这里的 `-m "..."` 是你本次提交的说明，之后方便理解该提交是干什么的。

大家好不好其，这里的一次提交到底是一个什么东西呢？

在 Git 里，一个 **commit** 可以理解成一次保存进度的 **快照**，Git 会给这次快照一个永久的编号（哈希），以后我们可以根据哈希回到这份状态。

我们来看看 ChatGPT 对一次提交的解释：

>一次 commit 本质上是 Git 数据库里的一个对象，里面主要记录了：
>
>1. **项目文件的快照指针**（指向一个 tree：当时目录/文件结构的快照）
>2. **父提交是谁**（上一条 commit 的哈希；首个提交没有父提交）
>3. **作者/提交者信息**（名字、邮箱、时间）
>4. **提交说明 message**（比如 “fix bug”）
>
>所以 commit 不是只保存改动，你可以把它理解成 **对整个仓库内容做了一个可追溯快照**（内部做了很多去重，所以并不会每次都拷贝一份全量文件）。
>
>从改代码到 commit 的过程是什么？
>
>Git 有几个区域（最重要的三块）：
>
>1. **工作区（Working Directory）**：你正在编辑的文件
>2. **暂存区（Index / Staging Area）**：你挑选出来准备提交的内容
>3. **仓库（Repository）**：已经提交的历史（commit 链）
>
>你执行命令时发生的事：
>
>① 你修改文件（工作区变脏）
>
>你改了 `a.txt`，此时只是工作区变了，Git 还没把它算进历史。
>
>② `git add a.txt`（放进暂存区）
>
>把你选择的改动放进准备提交的清单里。暂存区就像这次提交我打算带哪些改动。
>
>③ `git commit -m "xxx"`（生成提交对象）
>
>这一步会：
>
>- 把暂存区的内容生成一个 **快照树（tree）**
>- 创建一个 **commit 对象**，写入：
> - tree 的指针
> - 父 commit（上一条）
> - 作者信息、时间、message
>- 然后让当前分支（比如 `dev-han`）指向这个新的 commit
>- `HEAD` 仍然指向当前分支

我觉得理解这些知识需要适当的了解 Git 到底是什么，Git 的底层到底是什么。这部分内容当然是由我们亲爱的 ChatGPT 来回答一下：

> Git 是一个 **分布式版本控制系统**：用来记录你项目文件随时间的变化，让你能 **回退、对比、分支协作、合并**，并且每个人本地都有一份完整历史。
>
> #### Git 的底层本质：一个内容寻址的对象数据库 + 一套引用系统
>
> 即：**Git 底层就是把文件内容变成一堆对象（objects）存进 `.git/objects`，用哈希当地址；再用分支/标签/HEAD 这些引用（refs）去指向这些对象。**
>
> ##### 1）内容寻址（Content-addressed）
>
> Git 存东西不是按文件名/路径当主键，而是按 **内容的哈希** 当主键（传统是 SHA-1，新版本也可用 SHA-256）。
>
> 好处：
>
> - 内容不变，哈希不变 → 天然去重（同样内容只存一份）
> - 任何篡改都会导致哈希变化 → 完整性校验
>
> #### Git 主要存哪几种对象（4 大类）
>
> 这些对象都在 `.git/objects/` 里（新仓库很少，提交后会越来越多）。
>
> ##### A. blob：文件内容
>
> 一个文件的内容本体就是一个 blob（不含文件名）。
>
> #### B. tree：目录结构
>
> tree 记录：
>
> - 文件名 → 对应的 blob
> - 子目录名 → 对应的 tree
> - 文件权限（可执行位等）
>
> 所以 tree 就是某一刻目录长什么样的结构快照。
>
> ##### C. commit：一次提交
>
> commit 记录：
>
> - 指向一个 tree（这一刻的项目快照）
> - parent commit（上一条提交）
> - author/committer、时间、message
>
> commit 串起来就是历史链。
>
> ##### D. tag：标签（可选）
>
> tag 可以给某个 commit 起一个固定名字（比如 `v1.0.0`），可带签名。
>
> #### 分支、HEAD、tag 底层是什么？
>
> 它们不是对象本体，而是 **引用（refs）**——本质就是一些小文件，里面写着一个 commit 哈希。
>
> - 分支：`.git/refs/heads/dev-han` 内容是一个 commit hash
> - 远端跟踪分支：`.git/refs/remotes/origin/dev-han`
> - 标签：`.git/refs/tags/v1.0.0`
> - HEAD：`.git/HEAD`（通常指向某个分支引用：`ref: refs/heads/dev-han`）
>
> 所以：
>
> - **分支只是指向 commit 的指针**
> - commit 才是真正的历史节点
>
> #### Git 怎么做到“快照”却不浪费空间？
>
> 直觉上快照很大，但 Git 有两层关键优化：
>
> 1. **内容去重**：同内容的 blob/tree 只存一次（哈希相同就复用）
> 2. **打包压缩（packfiles）**：很多对象会被打成 `.pack`，并用差异存储（delta）压缩
>
> 因此提交很多次也不会线性爆炸。
>
> #### 工作区 / 暂存区 / 仓库：底层分别是什么
>
> 你平时接触的三个区：
>
> - **工作区**：你磁盘上的项目文件
> - **暂存区（index）**：`.git/index` 文件（记录下一次要提交的快照长什么样）
> - **仓库（history）**：`.git/objects` 里的对象 + `.git/refs` 里的引用
>
> `git add`：把工作区内容写成 blob，并更新 index
>  `git commit`：把 index 变成 tree，再生成 commit 对象，并移动分支指针

阅读完上面的内容，不知道大家好不好奇什么 Git 要把内容和目录分开存储，带着这个疑问，我们可以继续问 ChatGPT：

> Git **刻意把文件内容和目录结构分开存**，这是它底层设计的关键点之一。
>
> #### 为什么要分开：Git 的两层抽象
>
> ##### 1) 内容（blob）：只管内容，不管名字
>
> - **blob 只保存文件内容**（字节流）
> - **不保存文件名、路径、时间戳**
> - 同样内容 → 同一个 blob（天然去重）
>
> 举个例子：
>  `a.txt` 和 `copy.txt` 内容完全一样 → Git 底层可以只存 **一个 blob**。
>
> ##### 2) 目录结构（tree）：只管名字/层级/权限，不管内容本身
>
> - tree 里保存的是一堆条目：
>   - 文件名 → 指向某个 blob
>   - 子目录名 → 指向某个 tree
>   - 权限（普通文件/可执行/子模块等）
> - 所以 tree 就像某一刻的文件系统目录快照。
>
> #### 这么设计带来的直接好处
>
> ##### A. 极强的去重能力
>
> - 内容相同的文件，无论出现在哪个路径、叫啥名字，只存一份 blob。
> - 很多目录结构也能复用：某个子目录没变，对应的 tree 也能复用。
>
> ##### B. 重命名/移动变得便宜
>
> Git 不需要专门存 rename 记录才能支持重命名：
>
> - 你把 `a.txt` 改名为 `b.txt`，内容没变：
>   - blob 不变（同一个内容哈希）
>   - 只是 tree 里把文件名条目从 `a.txt` 换成 `b.txt`
> - 所以重命名在底层其实就是目录映射变了。
>
> ##### C. 提交就是指向一个 tree 的快照
>
> commit 只需要记录：
>
> - `tree`（这一刻目录长什么样）
> - `parent`（上一条提交）
>   就能把整套快照串起来。
>
> 直观类比，把项目想成：
>
> - **blob** = 书的内容页
> - **tree** = 目录/目录索引（第几章叫什么、指向哪些页）
> - **commit** = 这一版书的目录索引 + 上一版是谁 + 作者信息

到了这里，我们对 Git 应该有了一个整体的认知。

## git stash

有时候，项目有一个比较着急的 bug 让你切换到 `fix-han` 分支去修，但是你当前所在分支 `dev-han` 还有尚未完成的功能没有提交，这时执行 `git switch -c fix-han` 一定会报错，包括如果你现在执行 `git pull origin main` 合并远程仓库的代码。这是为什么呢？当然是因为你没有提交尚未修改的代码，Git 不允许这种操作存在。那么是不是我现在只要进行一次提交就可以了，当然可以。但如果你现在不想提交本次代码，那怎么做才能切换分支呢？

这时，`git stash -m "..."` 登场了！

`-m "..."` 同样是本次操作的说明，`git stash` 的作用是将当下尚未提交的代码隐藏起来，隐藏后，当前仓库就干净了，你就可以执行 `git switch` 操作了。等我们修复完 bug，想继续在原分支上开发，如何恢复被隐藏的代码？

首先，我们需要执行 `git stash list` 列出当前所有的隐藏，就像这样：

```bash
stash@{0}: On dev-han: first stash
```

然后查看一下我们之前写的代码是不是真的被隐藏了，嗯，确实没了。这时我们执行 `git stash apply "stash@{0}"`，再次查看代码，发现隐藏的代码出现了，是不是很神奇。再次执行 `git stash list`，发现 `stash@{0}` 还在，我们可以执行 `git stash drop stash@{0}`，这时刚刚的 `stash@{0}` 就不在了，如果有很多，可以执行 `git stash clear`，它会清除所有 `stash{n}`，其实实际使用时更推荐执行 `git stash pop stash{0}`，它会应用并删除 `stash{0}`。

第一次使用 `git stash` 系列命令，感觉这就像栈操作，事实上也确实如此。`git stash -m "..."` 全称是 `git stash push -m "..."`，一个 `stash{n}` 叫一个条目。

`git stash -m "..."` 默认只保存已跟踪文件，即执行过 `git add .`，但如果你想连未跟踪文件一起保存可以执行 `git stash -u -m "..."`。

其实当前工作空间有未提交内容时，执行 `git switch` 不一定会失败，只有切换的分支与当前分支文件的内容有冲突时才会被阻止，就像拉取远程分支默认合并代码时的规则类似。

为什么有冲突会被组织呢？

我的理解是，当工作空间有内容尚未提交，切换到另一个分支 Git 不做任何阻止，另一个分支的内容很可能会覆盖当前工作空间尚未提交的内容，由于尚未提交，我们无法追溯这部分代码，所以这部分代码就会被覆盖，导致代码丢失，你也不想刚写完的代码找不到吧？因此才要保存一下。

不知道大家有没有注意到，出现上面情况的前提是所有分支共享同一个工作空间和暂存区，事实上 Git 默认就是这样的。那么如果我们每个分支拥有独立的工作空间和暂存区的话，那是不是就不用执行 `git stash` 了？

是的，并且 Git 也给出了方案，就是 `git worktree`。这是 Git 的多工作目录功能，**让同一个仓库（同一个 `.git`）同时 挂出多个工作区**，每个工作区可以检出不同分支/提交，用来并行开发、修 bug、跑对比测试。但是我一个实习生，创建这么多分支开发不是纯扯淡了，所以实际没用过，只听说过。

## git reset

如果我们想要从当前的 commit 回退到之前的 commit 怎么办。

首先我们要执行 `git log` 罗列出我们所有的提交记录

```bash
commit 2544a7559e6231b6bb57ae5486e8065a5e8bfa59 (HEAD -> dev-han)
Author: hj <2978224828@qq.com>
Date:   Sat Feb 14 22:40:47 2026 +0800

    second commit

commit bba29710eb003d932fff730fbbb8052e34c2a898
Author: hj <2978224828@qq.com>
Date:   Sat Feb 14 21:30:50 2026 +0800

    first commit
```

`git log` 会显示一个提交的所有信息，如果觉得信息太多，可以执行 `git log --oneline`，他只会显示每次提交的一行，真实效果如其名 `one line`。

> 如果我们想要知道一个分支比远程分支多几个提交，可以执行 `git log origin/main..dev-han --oneline` 这条命令会显示当前分支 `dev-han` 比远程分支 `origin/main` 多几个提交

然后执行 `git reset bba29710eb003d932fff730fbbb8052e34c2a898`，这时我们就回到了这次提交。接下来详细解释一下回退的方式。先假设我们有如下提交历史：

```bash
A->B->C
```

如果我们执行 `git reset --soft <B>`

分支会回到 B，暂存区会保持 `reset` 前的内同，工作区不会退。

`git reset --mixed <B>`，我们直接执行 `git reset <B>` 就是这个效果，他会将分支移到 B，暂存区的内容也会回到 B，原来暂存区的内容取消暂存，工作区基本不动。

`git reset --hard <B>` 会将分支移到 B，暂存区回到 B，工作区也强制回到 B，本地改动会被清除。

> 嗯，实际上直接用 `git reset hash_code` 得了，其他的也记不住。。。

## GitHub

上面我也提到过远程分支这个名称，顾名思义，这个分支是远程的，他不在我们本地。Git 是一个本地代码管理系统，不需要联网我们就可以使用 Git 来完成我们自己的版本管理。但是实际工作生活中我们肯定是要和其他人共同开发一套系统，因此单单只是本地管理肯定是不够的，这就涉及到了团队的协作。GitHub 登场啦！

GitHub 是全球最大的开源代码托管平台，我们可以和其他人通过 GitHub 共同开发。

首先如果我们想要将本地代码推送至远程的 GitHub，首先需要执行 `git remote add origin https://github.com/<user>/<repo>.git`，将 GitHub 仓库设置为远端，或者是执行 `git remote add origin git@github.com:<user>/<repo>.git`。这时我们可以执行 `git remote -v` 检查一下当前仓库有哪些远端。

远端建立成功后就可以执行 `git push -u origin main`，`-u` 是为了把你本地分支和远端分支建立跟踪关系，之后在该分支想推送代码就可以直接执行 `git push` 了。

除了推送，我们还可以拉取，执行 `git pull origin main`。这样，远程的代码就被我们拉取到本地啦！在开发的时候，如果远程的代码和本地的代码没有冲突，那么会默认发生合并的，这样不需要显示的处理代码冲突，但是如果同一份代码，远程别人推送的时候修改了，你在本地也修改了，这时候就会发生合并冲突。

### 合并冲突

如果发生合并冲突了，代码中会有提示 

`<<<<<<< ======= >>>>>>>`，这时，我们只要讲需要的代码保留，然后再执行

```bash
git add .
git commit
```

上面讲的是合并代码远程代码，我们如何手动合并本地两个分支的代码呢？

首先执行 `git branch` 得知我们当前所在的分支是 `main` 分支，然后执行 `git merge dev-han`，这会合并 `dev-han` 分支的内容到 `main` 分支上。

当然我们也可以执行 `git rebase dev-han`，`rebase` 被广泛翻译成基变，他会将合并的新内容搬到原来提交的前面，使整个提交历史成为一条干净的直线。上面的 `git merge dev-han` 则会保留完整的分支合并记录。

`git pull origin main` 默认使用 `merge` 的方式讲远程代码合并到本地分支上，如果我们想要使用基变的方式，也可以执行 `git pull origin main --rebase`，这样你的合并记录就是一条干净的直线了！

`rebase` 方式发生冲突如何解决？执行以下命令：

```bash
git add .
git rebase --continue
```

> 其实我执行合并的时候，这两种方式感触最深的是 `merge` 方式会让我写一个合并记录，但是 `rebase` 不需要。

GitHub 是最大的开源平台，还有个 GitLab，一般公司会使用他来作为公司内部开发的托管平台，因为他可以自己部署，隐私性比较好吧。

其实我们在执行一些 Git 操作时，Git 会在终端给出一些提示，包括解决合并冲突 `git rebase --continue`、将某个文件修改 `git restore` 等等。今天的分享就到这里啦！
