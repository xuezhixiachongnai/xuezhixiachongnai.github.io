ES 在并发修改同一文档时，使用的是乐观锁机制，通过 `_seq_no` 和 `_primary_term` 来保证数据正确性。

> **乐观锁**：总是乐观的认为在当前时间段，数据不会被其他线程修改。数据会关联一个版本号，或者是时间戳。当线程提交写操作时时，会通过比较 version 和 timestamp 是否变化，如果没变，写操作成功。如果变了本次操作失败
>
> **悲观锁**：总是悲观的认为当前时间段内数据一定会被其他线程修改，因此会对当前操作的资源上锁。

并且，**无论是是执行 `update` 更新字段，还是使用 `index` 执行覆盖写，只要文档内容发生变化，都会导致版本相关字段发生变化。**

```json
{
  "_seq_no": 10,
  "_primary_term": 3
}
```

- `_seq_no`：主分片上的每一次操作都会递增，是一个严格递增的操作序列号。代表这是第几次修改操作
- `_primary_term`：代表当前分片的任期，每次主分片发生切换时都会加 1。它是为了防止旧主分片的写操作覆盖新主分片的数据

 在 ES 6.x 及以前，`_version` 是主要并发控制手段。但在 ES 7.x+ 后，官方不再推荐用 `_version` 做并发控制。

如果我们的业务允许写冲突。为了保证最大吞吐的话，每次直接修改文档数据即可

```http
POST index/_update/1
{
  "doc": {
    "count": 5
  }
}
```

也可以这样调用，让 ES 内部发现 `_seq_no` 冲突，自动重试

```http
POST index/_update/1?retry_on_conflict=3
```

如果该业务是强一致的话，我们可以执行 GET 操作获取 `_seq_no`、`_primary_term` 等数据

```http
POST index/_update/1?if_seq_no=10&if_primary_term=3
```

这样如果发生写冲突的话，会直接返回错误。
