我们在使用搜索引擎（如 Google）时，输入几个字符就能立刻看到提示词，这种体验**并不是某一个组件的功劳**。

> **搜索补全是一个典型的前端 + 后端 + 缓存 + Elasticsearch协同工作的系统工程。**

## 整体流程

```markdown
用户输入
  ↓
前端输入框（防抖 / 节流）
  ↓
后端补全接口
  ↓
缓存（内存 / Redis）
  ↓
Elasticsearch（补全查询）
  ↓
结果返回
  ↓
前端渲染候选词
```

### 1. 用户输入 & 前端控制

当用户在前端输入框中输入：

```markdown
elas
```

前端不会在每敲一个字母就立刻请求一次后端，否则体验和性能都会崩。

真是做法是：

- **防抖（debounce）**
  - 用户停止输入 100～300ms 才发请求
- 或 **节流（throttle）**
  - 每 200ms 最多发一次请求

**这是“搜索补全快”的第一层保障**

### 2. 前端请求后端补全接口

前端发请求，例如：

```http
GET /api/search/suggest?q=elas
```

此时：

- `elas` 只是一个 **前缀**
- 后端的目标只有一个： **尽快返回可能的补全词列表**

### 3. 后端反应

#### 首先是查缓存

优先顺序通常是：

1. 本地内存缓存（Caffeine / Guava）
2. Redis
3. Elasticsearch（兜底）

> 搜索补全是高频、重复、强实时请求。很多用户会输入相同前缀，缓存命中率极高。

如果缓存没有命中，才会调用 ES。

ES 做的事情非常单一，根据提供的前缀，返回 N 个候选词。

它并不关心：

- 前端怎么渲染

- 用户是否继续输入

- 请求频率控制

然后后端对 ES 结果做业务加工，比如：

- 去重

- 按热度排序

- 按业务规则过滤（黑词、低质量词）

- 限制返回数量（如最多 5～10 条）

### 4. 前端渲染补全列表

前端拿到结果后：

- 实时更新下拉列表
- 不会影响用户继续输入
- 用户输入变化 → 再次触发上述流程

我们在体验 Google 时非常流畅，这是为什么呢？

快的原因总结

1. **前端防抖 / 节流**
2. **短查询、短响应**
3. **缓存命中率极高**
4. **ES 补全查询是前缀匹配 / FST，复杂度极低**
5. **结果集非常小（通常 ≤ 10 条）**

搜索补全是一个**为低延迟而生的系统**

### 防抖

> 防抖是指在一段时间内，多次触发只执行最后一次。话句话说。只在用户停下来之后，才执行真正的操作。

#### 为什么需要防抖？

你在搜索框里输入：

```markdown
e → el → ela → elas → elast
```

如果**每敲一个字母就请求一次后端**：

- 网络请求暴增
- 后端压力大
- 响应顺序错乱
- 用户体验变差

防抖的目的就是：

> **等用户输入完一小段再请求**

#### 防抖的核心思想

假设防抖时间是 **300ms**

```markdown
时间轴 →
| e | el | ela | elas | elast |
    ↑   ↑    ↑
    每次输入都会“重新计时”
```

- 用户一直在输入就**不执行**
- 用户停止输入 ≥ 300ms 才会**执行一次**

#### 防抖的执行规则

规则总结

- 事件触发时开始计时
- 在计时结束前再次触发**清除计时器，重新计时**
- 直到一段时间内没有新触发执行回调

#### 典型使用场景

适合防抖的场景

- 搜索框自动补全
- 输入校验（用户名是否可用）
- 窗口 resize
- 表单实时校验
- 自动保存草稿

## ES 是如何实现提示词补全的？

当我们在 Google 输入框中输入：

```markdown
elast
```

Google 立刻给补全：

- elasticsearch

- elasticsearch tutorial

- elasticsearch vector search

**这本质上是根据已输入的前缀，快速找出最可能的完整词或短语**

在 Elasticsearch 中，有三种主流方式实现输入补全。

### `completion suggester`

这是 ES 官方为搜索提示设计的功能

#### 索引结构

```http
PUT suggest_index
{
  "mappings": {
    "properties": {
      "title_suggest": {
        "type": "completion"
      }
    }
  }
}
```

`title_suggest` 是用来存储补全提示词的，前端输入的前缀会从这里面找到对应的补全词。

#### 写入数据

```http
POST suggest_index/_doc/1
{
  "title_suggest": "elasticsearch vector search"
}
```

存入补全词

#### 查询补全

```http
POST suggest_index/_search
{
  "suggest": {
    "title-suggest": {
      "prefix": "elast",
      "completion": {
        "field": "title_suggest",
        "size": 5
      }
    }
  }
}
```

`suggest` 是和 `query` 同一级的作用域。就是用来查补全词的。`title-suggest` 是自定义的 `suggest` 的一个操作名。`prefix` 是要匹配的前缀。`completion` 指定这是一个 `completion` 类型的 `suggest`，`completion.field` 指定用哪个字段做补全（必须是 `type: completion` 的字段）；`completion.size` 表示最多返回几条候选词。

#### 返回结果

```json
{
  "took": 2,
  "timed_out": false,
  "suggest": {
    "title-suggest": [
      {
        "text": "elast",
        "offset": 0,
        "length": 5,
        "options": [
          {
            "text": "elasticsearch vector search",
            "_score": 1.0,
            "_id": "1"
          }
        ]
      }
    ]
  }
}
```

输出字段解释：

- `took`
  - 本次查询耗时（ms）
- `timed_out`
  - 是否超时
- `suggest`
  - suggest 返回结果的总入口
- `suggest.title-suggest`
  - 对应你请求里写的 suggest 名字
- `text`
  - 本次被补全的输入（原始前缀）
- `offset`
  - 前缀在输入文本中的起始位置（一般为 0）
- `length`
  - 前缀长度
- `options`
  - 候选项列表
- `options[].text`
  - 候选补全词条
- `options[]._score`
  - 候选的评分（completion 的评分机制和全文检索不同）
- `options[]._id`
  - 候选词条所属文档 id（可用于回源）

> 前端真正要展示的基本就是 `options[].text`。

### `edge_ngram`

这是最常使用的方案。它本质上不是 suggest API，而是**把前缀拆成很多 token 写进倒排索引**，然后用 `match` 查出来。因此，该方案不可避免的涉及到了分词器。

#### 索引结构

```http
PUT autocomplete_index
{
  "settings": {
    "analysis": {
      "tokenizer": {
        "edge_ngram_tokenizer": {
          "type": "edge_ngram",
          "min_gram": 1,
          "max_gram": 20,
          "token_chars": ["letter", "digit"]
        }
      },
      "analyzer": {
        "autocomplete_analyzer": {
          "tokenizer": "edge_ngram_tokenizer",
          "filter": ["lowercase"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "autocomplete_analyzer",
        "search_analyzer": "standard"
      }
    }
  }
}
```

我们只看对应实现的自定义分词器：

- `tokenizer.edge_ngram_tokenizer` 自定义一个 tokenizer

- `type: edge_ngram` 是内置的一种分词器，作用是把词拆成从开头开始的连续前缀片段。如 `elastic` 可能生成：`e`, `el`, `ela`, `elas`, ...

- `min_gram` 表示最短前缀长度（1 = 允许单字符）

- `max_gram` 表示最长前缀长度（20 = 最多生成 20 字符前缀）

- `token_chars` 限制参与分词的字符类型
- `letter`：字母
- `digit`：数字

> 支持中文，这里要换策略：中文不是 letter/digit

#### 查询

```json
{
  "match": {
    "title": "elast"
  }
}
```

`title` 中是需要匹配的前缀。返回值中

```json
{
  "took": 3,
  "hits": {
    "total": { "value": 1, "relation": "eq" },
    "hits": [
      {
        "_index": "autocomplete_index",
        "_id": "1",
        "_score": 1.23,
        "_source": {
          "title": "elasticsearch vector search"
        }
      }
    ]
  }
}
```

`_source` 是匹配内容。

### 搜索日志 + 热度统计

如何为用户生成热度最高的搜索内容？

做法：

1. 采集用户搜索词
2. 定期统计：
   - 搜索次数
   - 点击率
3. 将高频词写入 ES 的 suggest 索引

前端提交每一名用户的搜索内容。后端启用一个线程统计并分析该数据，然后将其写入 `completion` 补全词典中，其他用户在搜索相关内容使匹配该词典的补全词。因此我们看到的提示词，**不是预先设置号的字典，而是热搜**。

> 我们使用输入法时出现的内容是什么？它和 ES 搜索补全一样吗？
>
> 搜索引擎中的提示词与自动补全，本质上是基于前缀匹配和用户行为统计的搜索优化能力。Elasticsearch 提供了如 completion suggester 和 ngram 分词等机制，可高效实现搜索提示功能。而输入法的联想与纠错主要依赖语言模型和拼音转换算法，与搜索引擎属于不同技术体系，ES 只能在词库和热词层面提供辅助支持。
